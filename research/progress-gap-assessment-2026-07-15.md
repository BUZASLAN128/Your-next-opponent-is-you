## Answer First

The project is proceeding in the right **dependency order**, but the two kinds
of progress are very different:

- the privacy, provenance, modularity, no-action, cold-start, and experimental
  apparatus is materially ahead of an ordinary persona demo;
- the scientific evidence that the apparatus represents this user is still at
  the beginning.

The strongest honest status is:

> Ready for a separately authorized metadata and annotation pilot; not ready
> for a faithful-persona, virtual-self, autonomous learning, or AGI-like claim.

A single completion percentage would hide this distinction and would invent a
threshold the research has not calibrated. The matrix below therefore scores
apparatus and persona evidence separately.

## Assessment Method

This assessment compares the current source tree with Layers 0 through 8 in
the [convergence map](convergence-map.md). It distinguishes:

1. an implemented contract or safety mechanism;
2. a synthetic unit, integration, or acceptance check;
3. a real-user scientific result.

A first bounded observation now exists in the third category, but it is one
selected ten-atom correction chain rather than a persona evaluation. No
conversation export, real annotation set, real-person temporal holdout, or
user-audited fidelity result has been collected or run.
The V1, V1.1 atomic-review, and V1.2 decision-lifecycle checkpoints are
represented in the current public snapshot and draft PR 1. A first user-facing
declared-only calibration was reproduced in memory. One fresh private review
produced ten proposal-only atoms; the user reviewed all ten through two linked
canonical private receipts. This creates correction-loop evidence, not
evidence that the system represents the person accurately.

The final local engineering evidence is 272 passing tests with 82.00 percent
measured branch coverage, including real loopback PostgreSQL integration. That
number measures implementation regression coverage; it is not a persona
accuracy score and does not change the absence of real-user evidence.

## Established Apparatus Strengths

- **Evidence honesty:** output types expose unknowns, source receipts, data
  classes, and no-action truth instead of relying on model prose.
- **Privacy and authority:** private identity is separated from public code and
  D0-only external egress; persona confidence cannot grant authority.
- **Cold start:** absence of history yields a useful but explicitly generic
  Manager rather than a fabricated user model.
- **Engineering discipline:** responsibilities are split across small modules;
  source limits, strict typing, linting, compilation, unit, integration, and
  synthetic acceptance gates exist.
- **Decision lifecycle:** explicit user corrections can be replayed and
  explained through source and receipt identifiers without a model, database,
  last-write-wins rule, or silent persona promotion.
- **Bounded model role:** a live local model can propose structured atoms while
  exact-span, scope, correction, promotion, and authority decisions remain in
  deterministic code.
- **Experimental ordering:** the confirmed architecture remains a bounded V1
  baseline; graph systems, extra frameworks, automatic promotion, and action
  capability are not being added before evidence demands them.

## Open Scientific Evidence Gaps

- only one selected ten-atom interaction has a complete two-receipt user

- no rationale or evidence-demand metric implementation;

These are not cosmetic backlog items. They are the evidence required to know
whether the central representation is useful or wrong.

## Competing Hypotheses

### H1 — The project is following the research dependency order

**Status: supported.** Governance, evidence attribution, atomic interpretation,
target layers, workspace boundaries, authority, and evaluation apparatus are
being separated in the order described by the research. The interaction review
and correction lifecycle remain source-linked, explicit, non-persisted, and
unpromoted. Conflicts are exposed rather than resolved through recency or
repetition.

**Could be weakened by:** adding graph infrastructure, automatic promotion,
external actions, or broad persona claims before the annotation and holdout
gates.

### H2 — The project is prematurely claiming a persona

**Status: mostly rejected for public contracts, still an active interaction
risk.** The README and implementation record state that real-person validation
has not begun. A preview can nevertheless feel like a finished persona to a
human reader. Its output must therefore continue to say `declared_only`,
`low_unvalidated`, `not_persisted`, `authority=none`, and
`automatic_core_promotion=false`.

**Could be supported by:** generated personality summaries, unstated
generalization, assistant-authored claims, or omission of empty views and
unknowns.

### H3 — The project has overbuilt infrastructure before seeing data

**Status: materially supported.** The repository has a substantial storage,
policy, lineage, adapter, CLI, and benchmark surface while corpus evidence is
still zero. Some of that work is justified by privacy and experimental
integrity, but another infrastructure phase now has lower information value
than a small authorized annotation experiment.

**Could be weakened by:** showing that the existing apparatus immediately
supports a safe metadata inventory, annotation pilot, ablation, and temporal
holdout without another framework or schema expansion.

## Corrections Made During This Assessment

Source review found three apparatus problems that would have weakened later
evidence:

1. predictors received a full benchmark case object even though they did not
   currently read its hidden fields;
2. synthetic fixtures echoed each target label into their support channels;
3. retrieval selected one same-scope persona record even when lexical overlap
   was zero.

The implementation now creates a target-free predictor input, uses
non-target-echo support fixtures, and abstains when no positive lexical match
exists. These changes improve experimental isolation. They do not create
persona evidence.

The first persona-preview draft also exposed attribution and storage hazards.
The revised input requires explicit user speaker, represented-user claim
holder, adoption, source authority, identity-interpretation plane, and honest
synthetic classification. Real D3 files must be inside an explicit private
root, and the subject must match the scoped person. Arbitrary Markdown cannot
enter the real declaration path until a safe explicit-adoption format exists.
Because the existing database does not store all of those adoption fields,
real bootstrap and real replacement persistence are also blocked rather than
silently discarding provenance. Synthetic persistence remains available for
apparatus tests, but a synthetic subject cannot share identity records with a
private subject. Repository mutations serialize that boundary transactionally,
corrections must remain in the target data class, and inference fails closed on
mixed legacy subjects. Public inference has no unbound reader, inspection uses
a separate non-inference interface, and subject-lock contention fails fast for
whole-operation retry rather than waiting in a lock-order cycle. A real schema
migration is a later ask-first decision.

The first user correction card then exposed an interpretation-granularity
hazard: one sentence had been assigned one kind even when it contained several
modalities and destination layers. The atomic review now validates exact
character spans and separates literal normalization, inference, consequence,
speech act, modality, claim type, target layer, confidence dimensions, and null
reasons. It does not extract automatically, confirm a classification, persist a
record, or make the proposed all-in-one report schema authoritative.

V1.2 then exposed a second hazard: a digest-valid receipt is not necessarily a
semantically valid correction. Construction and replay therefore both
revalidate the action against the immutable source review. The resulting state
checks receipt-chain references and explicit supersession, while the decision
brief keeps source conversation, turn, target layer, scope, and correction
identifiers inspectable. This remains synthetic protocol evidence; hashes are
integrity checks, not signatures or identity authority.

The local extractor experiment then exposed a third boundary: schema-valid
model output is still only an untrusted proposal. Exact source occurrences,
scope inheritance, provider identity, and every no-promotion flag are therefore
revalidated outside the model. After the user authorized the outside-Git
review path, the first real attempt failed closed on a non-exact span and
wrote no review. Constraining model text to deterministic exact segments made
the next run structurally valid, but manual inspection found an interpretation
that extended beyond its cited source text and required correction. No
validated persona evidence was created.

The first genuine correction batch also exposed an assistant procedure error:
five actions were applied before approval. The user later approved A1-A5 and
chose to retain the result. The private receipt produced three split and two
rejected outcomes, preserved five pending atoms, and replayed twice to the same
state. The later ratification does not make the initial execution
pre-authorized. This is a useful negative signal about both the selected
extractor batch and the authorization protocol, but it is too small and
selectively chosen to estimate model quality or persona fidelity.

The user then approved the remaining five bounded actions through
a second receipt linked to the first. The complete state contains six split,
three rejected, one confirmed, and zero pending source atoms. Two full-chain
replays agreed; the 26-record deletion closure remained projection-only. Nine
of ten selected atoms required splitting or rejection, which strengthens the
correction-burden warning but still does not provide a randomized precision or
persona-fidelity estimate.

## Recommended Near-Term Sequence

1. Write a minimal branch-aware annotation guide and source-governance contract
   before requesting access to any conversation content.
2. Treat an official account export as a separate optional source and inventory
   it only after explicit authorization.
3. Freeze manual labels for a varied synthetic suite and compare the current
   Qwen3-8B pin with one pinned 1–3B local extractor. Both models may propose
   only and must preserve the same correction contract.
4. Compare raw retrieval, a static declared profile, and the structured core
   before adding consolidation.
5. Implement the missing rationale, evidence-demand, and calibration metric
   contracts.
6. Freeze a real chronological Mirror holdout and have the represented user
   audit attribution, judgment, scope, uncertainty, and abstention separately.
7. Change retrieval, schema, models, graph structure, or consolidation only in
   response to measured failure.

## Stop and Falsification Conditions

Revisit the representation before expanding capability if:

- humans cannot label the proposed fields consistently;
- the structured arm does not beat simpler baselines;
- wrong-person attribution or scope leakage appears;
- confidence remains poorly calibrated;
- corrections or deletion cannot propagate through lineage;
- the preview encourages overclaiming despite explicit uncertainty;
- privacy or authority boundaries fail under realistic local use.

The current result is therefore **research-aligned but scientifically early**.
The next valuable unit of progress is evidence from a small, authorized,
correctable user study—not another memory framework.
